# 내부 운영 기능 정비: 관리자 로그인, 신고 처리 알림, 운영 공지 자동화

서비스를 운영하다 보면 사용자 기능만큼이나 중요한 것이 운영 도구입니다. 이번 작업에서는 관리자 페이지 접근 방식, 신고 처리 흐름, 제재 알림, 티켓 조정 공지를 한 번에 정리했습니다. 목적은 단순했습니다. 운영자가 더 안전하게 로그인하고, 사용자는 자신의 신고나 제재 상황을 더 명확하게 안내받도록 만드는 것이었습니다.

## 관리자 로그인은 일반 사용자 로그인과 분리했습니다

이전에는 관리자 페이지에서도 일반 로그인 흐름을 그대로 재활용하는 방식이 섞여 있었습니다. 하지만 일반 사용자 로그인과 관리자 로그인을 같은 방식으로 다루면 운영 보안과 세션 정책을 따로 가져가기 어렵습니다.

이번에는 관리자 전용 로그인 엔드포인트를 분리하고, 내부 운영 계정을 기준으로 로그인할 수 있도록 정리했습니다. 운영 환경에서는 관리자 계정을 환경변수로 관리하고, 애플리케이션이 기동될 때 해당 계정이 없으면 자동 생성하고 있으면 최신 설정으로 동기화되도록 구성했습니다.

이 방식의 장점은 분명합니다.

- 관리자 로그인 정책을 사용자 로그인과 분리할 수 있습니다.
- 운영 계정의 권한을 명확하게 `ADMIN`으로 관리할 수 있습니다.
- 관리자 페이지 인증 흐름을 더 짧고 단순하게 가져갈 수 있습니다.

## 관리자 세션은 더 짧고 보수적으로 가져갑니다

관리자 계정은 일반 사용자보다 훨씬 강한 권한을 가집니다. 그래서 이번 정리에서는 관리자 로그인에 대해 refresh token을 제거하고 access token만 발급하도록 바꿨습니다.

즉, 관리자 페이지는 짧은 세션을 전제로 동작합니다. 조금 번거로울 수는 있지만, 운영 계정이 장시간 살아 있는 토큰을 들고 다니는 것보다 훨씬 안전합니다. 특히 내부 운영 도구는 "편의성보다 회수 가능성"이 더 중요한 경우가 많습니다.

## 신고 처리 상태가 사용자에게 바로 전달되도록 바꿨습니다

신고를 남긴 사용자는 종종 "내 신고가 접수만 된 건지, 실제로 검토되고 있는지, 기각된 건지"를 알 수 없어서 답답함을 느낍니다. 이번에는 관리자 페이지에서 신고 상태를 변경할 때 그 결과가 자동으로 공지성 알림으로 연결되도록 만들었습니다.

이제 운영자가 신고 상태를 `검토중`으로 바꾸면 신고자에게 검토 진행 안내가 전달됩니다. `처리 완료`로 바꾸면 처리 결과와 사유가 함께 안내되고, `기각`으로 바꾸면 기각 사유가 포함된 알림이 전달됩니다.

이 변화는 단순한 UX 개선을 넘어섭니다. 운영팀 입장에서도 별도로 메시지를 작성하거나 다른 공지 수단을 찾지 않아도 되고, 사용자 입장에서는 자신의 신고가 실제로 관리되고 있다는 신뢰를 느낄 수 있습니다.

## 제재와 매칭 제한도 운영 공지로 명확하게 안내합니다

사용자 제재는 실제 조치보다 안내 방식이 더 큰 불만으로 이어질 때가 많습니다. 아무 설명 없이 기능이 막히면 사용자는 오류인지 제재인지조차 알기 어렵기 때문입니다.

이번 변경으로 운영자가 특정 사용자에게 정지나 매칭 제한을 적용하면, 해당 사용자에게 조치 사유와 기간이 포함된 운영 공지가 자동으로 전달되도록 구성했습니다. 덕분에 사용자는 현재 상태를 명확히 이해할 수 있고, 운영팀도 반복 문의를 줄일 수 있습니다.

운영 관점에서는 이런 공지가 중요합니다. 제재는 "막는 것"에서 끝나는 것이 아니라 "왜 이런 상태인지 설명하는 것"까지 포함되어야 실제 운영 품질이 올라가기 때문입니다.

## 티켓 조정도 이제 운영 이력처럼 안내됩니다

운영 중에는 보상, CS 대응, 이벤트, 예외 복구 같은 이유로 특정 사용자에게 티켓을 수동 지급하거나 차감해야 하는 경우가 생깁니다. 그런데 사용자는 숫자만 바뀌면 그 이유를 알 수 없고, 운영팀은 나중에 다시 설명해야 하는 상황이 자주 발생합니다.

이번에는 운영자가 티켓을 조정할 때 지급 또는 차감 수량과 사유가 해당 사용자에게 자동으로 전달되도록 연결했습니다. 사용자는 왜 티켓이 늘었거나 줄었는지 바로 알 수 있고, 운영팀은 별도 공지나 안내 메시지를 추가로 보낼 필요가 없습니다.

이런 작은 차이가 쌓이면 운영 효율이 확실히 좋아집니다. 단순히 데이터만 수정하는 도구보다, 사용자 커뮤니케이션까지 함께 처리하는 운영 도구가 훨씬 강력하기 때문입니다.

## 관리자 기능은 더 숨기고, 사용자 기능은 그대로 열어두는 방향을 택했습니다

관리자 페이지와 관리자 인증 API는 일반 사용자 기능과 성격이 완전히 다릅니다. 그래서 운영용 경로는 네트워크 레벨에서 더 보수적으로 보호하는 방향을 권장하고 있습니다.

핵심 아이디어는 간단합니다.

- 관리자 로그인과 관리자 API는 가능한 한 외부에 직접 노출하지 않습니다.
- 필요하면 리버스 프록시나 IP 제한 같은 방식으로 운영 경로만 따로 보호합니다.
- 반대로 일반 사용자 API와 심사용 흐름은 그대로 열어두어야 합니다.

즉, 모든 것을 잠그는 것이 아니라 운영 도구만 더 강하게 감싸는 방식입니다. 이 구분이 명확해야 서비스 운영과 외부 심사, 두 가지를 함께 만족시킬 수 있습니다.

## 이번 정리의 핵심은 운영 경험을 제품 경험처럼 다룬 것입니다

이번 작업은 새로운 사용자 기능을 추가한 것은 아닙니다. 하지만 실제 서비스 품질에 미치는 영향은 꽤 큽니다. 관리자 로그인이 더 안전해졌고, 신고 처리와 제재, 티켓 조정이 모두 사용자 안내와 연결되면서 운영 흐름이 훨씬 일관되게 바뀌었습니다.

운영 도구는 보통 뒤로 밀리기 쉽지만, 서비스가 커질수록 운영 품질이 곧 사용자 경험이 됩니다. 누군가 신고를 넣었을 때, 제재를 받았을 때, 티켓이 지급되었을 때 "무슨 일이 일어났는지"를 명확히 전달하는 시스템은 생각보다 큰 신뢰 차이를 만듭니다.

이번 정리는 그 기본을 다지는 작업이었습니다.
