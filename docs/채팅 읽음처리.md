# 채팅 읽음처리 프론트 연동 가이드

이 문서는 현재 백엔드 구현 기준으로 프론트가 읽음처리를 어떻게 붙여야 하는지 정리한 문서다.

핵심 원칙은 하나다.

- 읽음 수(`unreadCount`)는 프론트가 계산하지 말고, 서버가 내려준 값을 그대로 신뢰한다.

## 1. 백엔드 동작 요약

현재 읽음 동기화 진입점은 아래 3개다.

- STOMP `SUBSCRIBE /sub/chat/room/{roomId}`
- `PATCH /api/v1/chat/rooms/{roomId}/read`
- `GET /api/v1/chat/rooms/{roomId}/messages`

읽음처리 전략은 방 타입별로 다르다.

- `PERSONAL`
- `GROUP`

또한 현재 백엔드는 "해당 방을 실시간으로 구독 중인 사용자"를 그 방을 보고 있는 사용자로 취급한다.

즉 프론트는 실제로 보고 있는 방만 구독해야 한다.

- 방 화면 진입: 해당 방 `SUBSCRIBE`
- 방 화면 이탈: 반드시 `UNSUBSCRIBE`
- 탭 전환, 모달 닫기, 다른 방 이동: 기존 방 `UNSUBSCRIBE`

이걸 지키지 않으면 실제로 안 보고 있는데도 읽음으로 처리될 수 있다.

## 2. 프론트가 반드시 따라야 하는 룸 진입 순서

채팅방 화면을 열 때 권장 순서는 아래다.

1. STOMP로 `/sub/chat/room/{roomId}` 구독
2. 바로 `GET /api/v1/chat/rooms/{roomId}/messages` 호출
3. 필요하면 fallback으로 `PATCH /api/v1/chat/rooms/{roomId}/read` 호출

설명:

- `SUBSCRIBE`가 실시간 읽음처리의 primary path다.
- `GET /messages`도 읽음 sync를 수행하므로, 실제 방 내용을 열 때 unread가 맞춰진다.
- `PATCH /read`는 websocket subscribe 지연, reconnect, 복구 실패 상황의 fallback이다.

권장 사용 시점:

- 웹소켓 재연결 직후
- 앱 foreground 복귀 직후
- `SUBSCRIBE` 성공 여부가 불확실한 경우

남용하지 말아야 하는 시점:

- 스크롤할 때마다
- 메시지 한 개 받을 때마다
- `READ_RECEIPT` 받을 때마다

## 3. 사용해야 하는 STOMP/API 경로

실시간 수신:

```text
SUBSCRIBE /sub/chat/room/{roomId}
```

실시간 발신:

```text
SEND /pub/chat/message
```

HTTP 메시지 조회:

```text
GET /api/v1/chat/rooms/{roomId}/messages?lastMessageId={optional}&size={optional}
```

HTTP 읽음 fallback:

```text
PATCH /api/v1/chat/rooms/{roomId}/read
```

## 4. 프론트가 처리해야 하는 이벤트 타입

현재 프론트가 읽음처리 관점에서 봐야 하는 이벤트는 2개다.

- `MESSAGE`
- `READ_RECEIPT`

### 4.1 `MESSAGE`

새 메시지 이벤트다.

중요 규칙:

- `MESSAGE.unreadCount`는 이미 서버가 계산한 최종 초기 unread 값이다.
- 프론트는 참여자 수로 다시 계산하면 안 된다.
- 특히 그룹방에서는 이미 보고 있는 멤버 수가 빠진 값으로 올 수 있다.
- 1:1에서도 상대가 이미 보고 있으면 처음부터 `0`으로 온다.

예시:

```json
{
  "eventType": "MESSAGE",
  "messageId": 101,
  "roomId": 55,
  "senderId": 7,
  "content": "안녕",
  "messageType": "TEXT",
  "deleted": false,
  "unreadCount": 0,
  "readAt": "2026-04-17T06:10:12.123456Z",
  "sentAt": "2026-04-17T06:10:12.123456Z",
  "createdAt": "2026-04-17T06:10:12.123456Z"
}
```

프론트 처리 규칙:

- `messageId` 기준으로 메시지 upsert
- `unreadCount`는 payload 값으로 그대로 저장
- `readAt`도 payload 값으로 그대로 저장

### 4.2 `READ_RECEIPT`

기존 메시지의 unread 수가 줄었음을 알리는 이벤트다.

예시:

```json
{
  "eventType": "READ_RECEIPT",
  "messageId": 101,
  "roomId": 55,
  "unreadCount": 2,
  "readAt": "2026-04-17T06:11:05.000000Z"
}
```

프론트 처리 규칙:

- `messageId`로 기존 메시지를 찾는다
- 해당 메시지의 `unreadCount`를 payload 값으로 덮어쓴다
- `readAt`는 `unreadCount === 0`일 때만 메시지의 최종 `readAt`으로 반영한다
- `unreadCount > 0`이면 `READ_RECEIPT.readAt`는 "이번 읽음 sync 시각"일 뿐이므로, 메시지의 최종 읽음 시각으로 쓰지 않는다

중요:

- `READ_RECEIPT`를 받았다고 프론트가 `currentUnread - 1` 식으로 계산하면 안 된다
- 반드시 서버가 보낸 `unreadCount` 값으로 덮어써야 한다

## 5. `readAt` 의미

`readAt`은 이벤트 종류에 따라 의미가 다르다.

### 5.1 `MESSAGE.readAt`

- 그 메시지가 완전히 읽혀서 `unreadCount == 0`이 된 시각
- 그룹채팅에서는 마지막 미읽음 멤버까지 읽은 시각
- 아직 다 안 읽혔으면 `null`일 수 있다

### 5.2 `READ_RECEIPT.readAt`

- 이번 읽음 동기화가 발생한 시각
- 메시지가 완전 읽힘이 아닐 수도 있다

프론트 UI 규칙:

- 말풍선 하단의 최종 읽음 시각 표시에는 `MESSAGE.readAt` 또는 `READ_RECEIPT`에서 `unreadCount == 0`일 때의 `readAt`만 사용
- 중간 감소 단계(`3 -> 2`, `2 -> 1`)에서는 `readAt`을 최종 읽음 시각처럼 노출하지 않는다

## 6. 방 타입별 프론트 기대 동작

### 6.1 1:1 채팅

기대 동작:

- 상대가 방에 들어오면 내 기존 메시지의 `unreadCount`가 `1 -> 0`으로 바뀐다
- 상대가 이미 방을 보고 있으면 내가 보내는 새 메시지는 처음부터 `unreadCount == 0`으로 온다

프론트 구현 포인트:

- 1:1에서 unread badge를 직접 누적 계산하지 않는다
- 메시지별 `unreadCount`는 사실상 `0` 또는 `1`로 취급하면 된다
- 새 메시지에 대해 항상 `READ_RECEIPT`가 추가로 오는 것은 아니다
- 상대가 이미 보고 있으면 `MESSAGE` 한 번만 오고, 그 안에 이미 `unreadCount: 0`이 들어 있다

### 6.2 그룹 채팅

기대 동작:

- 기본 unread는 `sender 제외 + 현재 unread 대상 수`로 시작한다
- 멤버가 읽을 때마다 `3 -> 2 -> 1 -> 0`처럼 줄어든다
- 이미 그 방을 보고 있는 멤버는 새 메시지 unread 계산에서 처음부터 제외될 수 있다

프론트 구현 포인트:

- 새 메시지의 초기 unread도 서버 값을 그대로 사용
- `READ_RECEIPT`가 올 때마다 해당 메시지의 unread를 덮어쓰기
- 참여자 수, hidden 멤버 수, 현재 viewing 멤버 수를 프론트에서 따로 계산하지 않는다

## 7. unread 대상이 아닌 메시지

아래 메시지는 unread 추적 대상이 아니다.

- `deleted == true`인 메시지
- `messageType`이 `TEXT`, `IMAGE`가 아닌 메시지
- 예: `ENTER`, `EXIT`, 시스템 메시지

프론트 처리 규칙:

- 이런 메시지는 unread badge 계산에 포함하지 않는다
- 읽음 숫자 UI가 있더라도 `0`으로 취급한다

## 8. 프론트 상태 모델 권장안

메시지 상태는 아래 정도로 들고 있으면 충분하다.

```ts
type ChatMessageState = {
  messageId: number
  roomId: number
  eventType: "MESSAGE" | "READ_RECEIPT"
  content: string | null
  messageType: string | null
  deleted: boolean
  unreadCount: number
  readAt: string | null
  sentAt: string | null
  createdAt: string | null
}
```

권장 normalize:

```ts
const normalizedMessageId = payload.messageId ?? payload.id
const normalizedRoomId = payload.roomId ?? payload.chatRoomId
```

이유:

- 현재 응답에는 호환성 때문에 `id/messageId`, `roomId/chatRoomId`가 같이 들어올 수 있다

## 9. 최소 처리 로직 예시

```ts
function handleChatEvent(payload: ChatMessagePayload) {
  const messageId = payload.messageId ?? payload.id

  if (payload.eventType === "MESSAGE") {
    upsertMessage(messageId, {
      ...existingMessage(messageId),
      ...payload,
      unreadCount: payload.unreadCount ?? 0,
      readAt: payload.readAt ?? null,
    })
    return
  }

  if (payload.eventType === "READ_RECEIPT") {
    const prev = getMessage(messageId)
    if (!prev) return

    upsertMessage(messageId, {
      ...prev,
      unreadCount: payload.unreadCount ?? prev.unreadCount,
      readAt: payload.unreadCount === 0 ? payload.readAt ?? prev.readAt : prev.readAt,
    })
  }
}
```

## 10. room list unread 처리

`GET /api/v1/chat/rooms`의 `ChatRoomResponse.unreadCount`를 그대로 사용하면 된다.

현재 백엔드 기준:

- `PERSONAL` 방 목록 unread는 `0` 또는 `1`
- `GROUP` 방 목록 unread는 실제 unread 메시지 수

현재 보고 있는 방을 열었을 때는 아래 둘 중 하나로 room list badge를 바로 0으로 맞추면 된다.

- room list를 다시 조회
- 로컬 상태에서 현재 방 unread badge만 0으로 갱신

## 11. 시간값 처리

응답의 시간값은 UTC offset 포함 문자열이다.

- `readAt`
- `sentAt`
- `createdAt`

프론트 규칙:

- 서버 시간 문자열은 그대로 파싱
- 화면 표시만 사용자 로컬 타임존으로 변환
- 문자열 재조합으로 직접 timezone 계산하지 않는다

## 12. 프론트에서 하지 말아야 할 것

- 참여자 수로 unread를 직접 계산하지 않기
- `READ_RECEIPT`마다 `-1` 감소시키지 않기
- 실제로 안 보고 있는 방을 계속 subscribe 상태로 두지 않기
- `READ_RECEIPT.readAt`을 항상 최종 읽음 시각으로 쓰지 않기
- 1:1 채팅에서 "새 메시지면 무조건 unread 1"이라고 가정하지 않기

## 13. 프론트 체크리스트

- 채팅방 진입 시 `/sub/chat/room/{roomId}` subscribe
- 채팅방 진입 직후 `GET /api/v1/chat/rooms/{roomId}/messages`
- reconnect/복구 시 `PATCH /api/v1/chat/rooms/{roomId}/read` fallback 사용
- 채팅방 이탈 시 반드시 `UNSUBSCRIBE`
- `MESSAGE.unreadCount`를 그대로 반영
- `READ_RECEIPT`는 `messageId` 기준으로 `unreadCount` 덮어쓰기
- `READ_RECEIPT.readAt`은 `unreadCount == 0`일 때만 최종 읽음 시각으로 반영

이 규칙대로 붙이면 현재 백엔드 읽음처리와 프론트 상태가 가장 안정적으로 맞는다.
