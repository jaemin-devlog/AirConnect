# AirConnect 클라이언트 수정 가이드

## 변경 목적

이제 학교 이메일 인증은 필수 온보딩 단계가 아니라 선택 보상 미션이다.

서버 정책은 아래처럼 바뀌었다.

- 학교 이메일 미인증이라고 해서 핵심 API 접근을 막지 않는다.
- 프로필 사진 등록 시 티켓 1장 지급
- 학교 이메일 인증 시 티켓 1장 지급

즉, 클라이언트도 `강제 입력 플로우`에서 `선택 보상 플로우`로 바꿔야 한다.

## 핵심 변경 요약

- `emailVerified == false` 만으로 시작 화면, 온보딩 화면, 차단 화면으로 보내지 않는다.
- 학교 이메일 인증 화면은 필수 단계가 아니라 선택 미션 화면으로 노출한다.
- 프로필 사진 등록도 필수 단계가 아니라 선택 미션으로 노출한다.
- `사진 등록 1장`, `이메일 인증 1장` 보상을 UI에 명확히 보여준다.
- 로그인 직후 흐름에서 이메일 인증 강제 라우팅을 제거한다.

## 서버 기준 동작

서버는 이제 학교 이메일 미인증 유저도 보호 API 접근을 허용한다.

- 더 이상 `SCHOOL_EMAIL_VERIFICATION_REQUIRED` 차단을 전제로 라우팅하지 않는다.
- `GET /api/v1/users/me` 에서 아래 상태값을 계속 내려준다.
  - `onboardingStatus`
  - `profileExists`
  - `profileImageUploaded`
  - `emailVerified`
  - `tickets`

클라이언트는 이 값을 `접근 제어`가 아니라 `미션/보상 상태 표시` 용도로 사용해야 한다.

## 라우팅 수정

### 제거해야 하는 것

- 로그인 직후 `emailVerified == false` 면 이메일 인증 화면으로 강제 이동하는 로직
- 홈 진입 전 학교 이메일 인증이 끝나야만 다음 화면으로 가는 로직
- 매칭, 채팅, 메인 탭 진입 시 학교 이메일 인증 여부만으로 막는 가드
- 프로필 사진이 없다고 전체 앱 흐름을 막는 가드

### 유지하거나 조정할 것

- 기본 가입 정보 입력이 정말 서비스상 필수라면 그 단계는 유지 가능
- 다만 학교 이메일 인증과 프로필 사진 등록은 필수 단계로 두지 않는다
- 인증/사진 등록은 홈, 마이페이지, 미션 배너, 보상 카드 등에서 나중에 수행 가능해야 한다

## 추천 UX 흐름

### 로그인 직후

- 소셜 로그인 성공
- 필요한 최소 가입 정보가 있으면 해당 단계만 진행
- 가입 완료 후 바로 메인 경험 진입
- 메인 또는 마이페이지에 선택 미션 카드 노출
  - 프로필 사진 등록하고 티켓 1장 받기
  - 학교 이메일 인증하고 티켓 1장 받기

### 마이페이지 또는 홈 미션 섹션

- `profileImageUploaded == false` 이면 사진 등록 미션 노출
- `emailVerified == false` 이면 학교 이메일 인증 미션 노출
- 둘 다 완료되면 완료 상태와 받은 보상을 표시

## 상태값 해석 규칙

### `onboardingStatus`

- `BASIC`: 기본 가입 정보만 있는 상태
- `FULL`: 가입/프로필 입력이 완료된 상태

주의:

- `emailVerified` 와 `onboardingStatus` 는 같은 의미가 아니다
- `emailVerified == false` 여도 정상 사용자로 앱 사용이 가능하다

### `profileExists`

- 프로필 상세 정보 존재 여부
- 이 값이 `false` 라고 해서 이메일 인증으로 보내면 안 된다

### `profileImageUploaded`

- 프로필 사진 보상 미션 완료 여부
- `false` 면 “티켓 1장 받기” CTA 표시

### `emailVerified`

- 학교 이메일 인증 보상 미션 완료 여부
- `false` 면 “티켓 1장 받기” CTA 표시

## 보상 UI 수정

기존에 `사진 2장`, `이메일 0장` 또는 필수 플로우 전제 UI가 있다면 아래로 바꾼다.

- 프로필 사진 등록 보상: 티켓 1장
- 학교 이메일 인증 보상: 티켓 1장

추천 문구:

- `프로필 사진 등록하고 티켓 1장 받기`
- `학교 이메일 인증하고 티켓 1장 받기`

완료 상태 문구 예시:

- `프로필 사진 등록 완료`
- `학교 이메일 인증 완료`

## API 사용 가이드

### 사용자 상태 조회

- `GET /api/v1/users/me`

이 응답으로 아래를 판단한다.

- 현재 사용자 정보
- `profileImageUploaded`
- `emailVerified`
- 현재 티켓 수

### 학교 이메일 인증

- `POST /api/v1/verification/email/send`
- `POST /api/v1/verification/email/verify`

인증 성공 시:

- 보상 완료 토스트 또는 배너 표시
- `GET /api/v1/users/me` 재조회 또는 로컬 상태 갱신
- `emailVerified = true`
- 티켓 1장 증가 반영

### 프로필 사진 업로드

- `POST /api/v1/users/profile-image`

업로드 성공 시:

- 보상 완료 토스트 또는 배너 표시
- `GET /api/v1/users/me` 재조회 또는 로컬 상태 갱신
- `profileImageUploaded = true`
- 티켓 1장 증가 반영

## 클라이언트 로직 체크리스트

- `emailVerified == false` 강제 라우팅 제거
- 학교 이메일 미인증 상태에서도 메인 화면 진입 허용
- 학교 이메일 인증을 선택 미션으로 배치
- 프로필 사진 등록을 선택 미션으로 배치
- 보상 수량을 `1 / 1`로 수정
- 보상 완료 후 티켓 수 즉시 반영
- `users/me` 기반 미션 완료 상태 동기화
- 완료된 미션은 다시 CTA를 띄우지 않도록 처리

## QA 체크리스트

- 학교 이메일 미인증 상태로도 로그인 후 메인 진입이 되는지
- 학교 이메일 미인증 상태에서 매칭/채팅/일반 기능 접근이 막히지 않는지
- 프로필 사진 등록 후 티켓이 1장 증가하는지
- 학교 이메일 인증 후 티켓이 1장 증가하는지
- 사진 등록 완료 후 해당 미션이 완료 상태로 바뀌는지
- 이메일 인증 완료 후 해당 미션이 완료 상태로 바뀌는지
- 기존에 이메일 인증 강제 화면으로 튀던 흐름이 사라졌는지
- 문구/배너/보상 숫자가 모두 `1장`으로 맞는지

## 한 줄 정리

클라이언트는 이제 학교 이메일 인증과 프로필 사진 등록을 `필수 온보딩`이 아니라 `선택 보상 미션`으로 처리해야 한다.
